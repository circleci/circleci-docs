== Run a workflow only when a pull request is opened

=== Goal
You may want to configure a workflow in your pipeline that will only run on a pipeline that is triggered when a pull request is opened. To do this you can use a workflow filter.

=== Steps

* Update your CircleCI config to include a workflow that you only want to run when a PR is opened.
+
[,yml]
----
workflows:
  workflow-pr-open:
    # Only run this workflow when a PR is opened
    when: pipeline.event.name == "pull_request" and pipeline.event.action == "opened"
    jobs:
      - my-job-1
      - my-job-2
----

=== Notes

* The `pipeline.event.name` and `pipeline.event.action` variables are only available for GitHub App pipelines. If you have a GitHub OAuth integration you can install the GitHub App into your GitHub org and then create GitHub App pipelines. This is a quick one-time action, as described in the xref:guides:integration:using-the-circleci-github-app-in-an-oauth-org.adoc[Using the CircleCI GitHub App in an OAuth Organization] page.

== Combine pipeline parameters with event filters in workflow conditions

=== Goal

You may want to configure a workflow that runs when a specific event occurs (such as a push or pull request) or when a pipeline parameter is set to `true`. To combine pipeline parameters with event filters in a workflow `when` clause, write the condition as a direct expression without `<< >>` delimiters.

The `when` clause accepts an expression as its argument. The expression is evaluated to a boolean value at configuration compilation time, before the workflow runs, so template replacement is not required. For more on using expressions in your configuration, see xref:reference:ROOT:configuration-reference.adoc#using-expressions-in-your-configuration[Using Expressions in Your Configuration].

=== Steps

. Define your pipeline parameters in the `parameters` section of your config file. In this example, `deploy` is used as the parameter name, but you can use any parameter name you define:
+
[,yaml]
----
version: 2.1

parameters:
  deploy:
    type: boolean
    default: false
----

. Write your workflow `when` clause as a direct expression. Reference pipeline parameters using `pipeline.parameters.<name>` directly (without `<< >>` delimiters):
+
[,yaml]
----
workflows:
  my-workflow:
    when: pipeline.event.name == "push" or (pipeline.event.name == "pull_request" and pipeline.event.github.pull_request.draft == false) or pipeline.parameters.deploy
    jobs:
      - build
----

=== Notes

* You must define your pipeline parameters in the `parameters` section before using them in workflow conditions. The parameter name `deploy` in this example is just an example. You can use any parameter name you define, for example, `run_tests`, `skip_validation`, or `enable_feature`.
* The workflow `when` clause expects a direct expression. Use pipeline parameter names directly (for example, `pipeline.parameters.deploy`).
* Do not mix `<< >>` delimiters with direct expressions. The `<< >>` syntax is used to mark expressions inside string values. If the `when` clause contains `<< >>` delimiters, the entire value is treated as a string with template replacement rather than as a direct expression. This causes the workflow to always be created regardless of the actual condition.
+
[source,yaml]
----
# Incorrect: mixing direct expression with << >> delimiters
workflows:
  my-workflow:
    when: pipeline.event.name == "push" or << pipeline.parameters.deploy >>
    jobs:
      - build
----
+
* For simple parameter checks, you can use `<< >>` syntax on its own, since the entire value is then a single expression. Remember to replace `deploy` with your own parameter name:
+
[source,yaml]
----
workflows:
  my-workflow:
    when: << pipeline.parameters.deploy >>
    jobs:
      - build
----
+
* For more details, see the xref:reference:ROOT:configuration-reference.adoc#using-when-in-workflows[Using When in Workflows] and xref:reference:ROOT:configuration-reference.adoc#using-expressions-in-your-configuration[Using Expressions in Your Configuration] sections in the configuration reference.

== Use no-op jobs to create a cleaner workflow graph

=== Goal

You may want to organize your workflow graph to make it more readable without adding additional build time or consuming credits. Using `no-op` jobs can help create a cleaner structure for complex workflows with many dependencies.

The following snippet shows how your workflow configuration might look without using a `no-op` job:

[,yaml]
----
version: 2.1

jobs:
  build:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm ci
      - run: npm run build

  test-unit:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run test:unit

  test-integration:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run test:integration

  test-e2e:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run test:e2e

  deploy:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run deploy

workflows:
  build-test-deploy:
    jobs:
      - build
      - test-unit:
          requires:
            - build
      - test-integration:
          requires:
            - build
      - test-e2e:
          requires:
            - build
      - deploy:
          requires:
            - test-unit
            - test-integration
            - test-e2e
----

Notice how the `deploy` job needs to explicitly list all three test jobs in its `requires` section. As your workflow grows, this list could become longer and more difficult to maintain.

Here is the workflow graph for this config:

.Workflow graph before adding no-op job
image::guides:ROOT:orchestrate-and-trigger/clean-graph-before.png[Workflow graph showing snippet without `no-op` job]

=== Steps

. Add a `no-op` job to your CircleCI configuration file.
+
[,yaml]
----
jobs:
  test-coordinator:
    type: no-op

  test-unit:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run test:unit

  test-integration:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run test:integration

  test-e2e:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run test:e2e
----

. Configure your workflow to use the `no-op` job as a coordinator for downstream jobs.
+
[,yaml]
----
workflows:
  build-test-deploy:
    jobs:
      - build
      - test-unit:
          requires:
            - build
      - test-integration:
          requires:
            - build
      - test-e2e:
          requires:
            - build
      - test-coordinator:
          requires:
            - test-unit
            - test-integration
            - test-e2e
      - deploy:
          requires:
            - test-coordinator
----

The workflow graph will now look like this:

.Workflow graph after adding no-op job
image::guides:ROOT:orchestrate-and-trigger/clean-graph-after.png[Workflow graph showing snippet with `no-op` job]

=== Notes

* The `no-op` job performs no actions and consumes no credits. This makes it an ideal structural element in your workflow.
* This pattern is useful when you want to visually group related jobs in the CircleCI UI.
* Using `no-op` jobs can simplify workflow management when dealing with complex dependency chains.
* Without the `test-coordinator` job, the `deploy` job would need to list all test jobs in its `requires` section.

== Simplify GitHub required status checks with no-op jobs

=== Goal

You may want to simplify the management of GitHub required status checks for your branch protection rules. Use a single `no-op` job as a gatekeeper that depends on all your critical checks. This approach is means you only need to manage a single required status check rather than multiple status checks in GitHub.

The following snippet shows how your workflow configuration might look without using a `no-op` job:

[,yaml]
----
version: 2.1

jobs:
  lint:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run lint

  test:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm test

  security-scan:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run security-check

workflows:
  main:
    jobs:
      - lint
      - test
      - security-scan
----

With this configuration, in GitHub you would need to set up branch protection rules for each job (`lint`, `test`, `security-scan`). You would need to update both CircleCI config and GitHub settings when adding more jobs.

=== Steps

. Add a `no-op` job to your CircleCI configuration file that will serve as your single required status check. In this example the no-op job is `all-tests-passed`:
+
[,yaml]
----
jobs:
  all-tests-passed:
    type: no-op

  lint:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run lint

  test:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm test

  security-scan:
    docker:
      - image: cimg/node:20.10
    steps:
      - checkout
      - run: npm run security-check
----

. Configure your workflow to make all critical jobs prerequisites for the `no-op` job. The `all-tests-passed` will only succeed if all its required jobs succeed.
+
[,yaml]
----
workflows:
  main:
    jobs:
      - lint
      - test
      - security-scan
      - all-tests-passed:
          requires:
            - lint
            - test
            - security-scan
----

. In GitHub, set up a branch protection rule to require only the `all-tests-passed` job as a required status check. See the link:https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches[GitHub docs] for more information on branch protection rules.

=== Notes

* This approach simplifies branch protection management in GitHub by requiring only one status check.
* When you add new checks, you only need to update the `requires` list for your `no-op` job.
* The `no-op` job will succeed only if all its required jobs succeed.
* This pattern helps keep GitHub and CircleCI configurations in sync without manual updates.

== Prevent deployment conflicts with serial groups

=== Goal

You may want to prevent deployment conflicts when multiple teams or projects need to deploy to the same environment. Using serial groups ensures that only one deployment runs at a time across your entire organization, preventing race conditions and ensuring deployments complete in the order they were triggered.

=== Steps

. Add the `serial-group` key to your deployment job in your CircleCI configuration file. Use a meaningful name that represents the environment being deployed to:
+
[,yml]
----
workflows:
  deploy-workflow:
    jobs:
      - test
      - build
      - deploy-staging:
          serial-group: staging-deployment
          requires:
            - test
            - build
----

. For multiple projects deploying to the same environment, use the same serial group value across all project configurations. This ensures sequential execution across your organization:
+
[,yml]
----
workflows:
  deploy-workflow:
    jobs:
      - test
      - build
      - deploy-staging:
          serial-group: << pipeline.project.slug >>/staging
          requires:
            - test
            - build
----

. Monitor the sequential execution in the CircleCI web app. Jobs in a serial group will show a key icon in the workflow graph and display a "queued" status when waiting for another job to complete.

=== Notes

* Serial groups work across your entire organization, not just within a single project or workflow.
* Jobs from newer pipelines (higher pipeline numbers) take priority over jobs from older pipelines.
* If a job waits in the queue for more than five hours, it will be automatically canceled.
* Using pipeline values like `<< pipeline.project.slug >>` in your serial group name allows you to create dynamic grouping patterns that can be shared across projects.
* For more information, see xref:controlling-serial-execution-across-your-organization.adoc[Controlling serial execution across your organization].

== Ensure database migrations run in sequence

=== Goal

You may want to ensure that database migrations run in the correct sequence across multiple services or projects. Using serial groups prevents concurrent migrations that could cause conflicts or data corruption.

=== Steps

. Add the `serial-group` key to your migration job. Use a serial group name that identifies the specific database or environment:
+
[,yml]
----
workflows:
  migration-workflow:
    jobs:
      - test
      - run-migration:
          serial-group: production-db-migrations
          requires:
            - test
----

. For multiple services that share a database, use the same serial group value in each service's configuration. This ensures migrations run one at a time:
+
[,yml]
----
workflows:
  service-workflow:
    jobs:
      - build
      - test
      - migrate-database:
          serial-group: shared-db/<< pipeline.git.branch >>
          requires:
            - test
      - deploy:
          requires:
            - migrate-database
----

. Add environment-specific serial groups to prevent migrations from different environments from interfering with each other:
+
[,yml]
----
workflows:
  deploy-workflow:
    jobs:
      - test
      - migrate-staging:
          serial-group: staging-db-migrations
          filters:
            branches:
              only: develop
      - migrate-production:
          serial-group: production-db-migrations
          filters:
            branches:
              only: main
----

=== Notes

* Each environment should have its own serial group to prevent cross-environment conflicts.
* Include environment or database identifiers in your serial group name to avoid accidental grouping.
* Serial groups maintain the lock even if a migration job fails, allowing the next migration in the queue to proceed.
* Pipeline number priority ensures newer migrations run before older ones, preventing out-of-order migrations.
* For more information, see xref:controlling-serial-execution-across-your-organization.adoc[Controlling serial execution across your organization].

== Control access to limited testing resources

=== Goal

You may want to control access to limited testing resources that can only handle one job at a time, such as physical devices, test environments with capacity constraints, or services with API rate limits. Using serial groups ensures only one job accesses the resource at any given time.

=== Steps

. Add the `serial-group` key to jobs that need access to the limited resource. Use a serial group name that identifies the specific resource:
+
[,yml]
----
workflows:
  test-workflow:
    jobs:
      - unit-tests
      - integration-tests:
          serial-group: test-environment-1
          requires:
            - unit-tests
----

. For multiple projects sharing the same resource, use the same serial group value across all configurations:
+
[,yml]
----
workflows:
  mobile-test-workflow:
    jobs:
      - build
      - device-tests:
          serial-group: physical-device-pool
          requires:
            - build
----

. Combine serial groups with workflow filters to control which branches access the limited resource:
+
[,yml]
----
workflows:
  test-workflow:
    jobs:
      - build
      - e2e-tests:
          serial-group: staging-environment
          filters:
            branches:
              only:
                - main
                - develop
          requires:
            - build
----

=== Notes

* Serial groups are scoped to your organization, ensuring resource protection across all projects.
* Jobs will wait in the queue for up to five hours before being automatically canceled.
* You can manually release a serial group from the CircleCI web app to allow the next job to start without canceling the current job.
* Consider using descriptive serial group names that clearly identify the resource being protected.
* For more information, see xref:controlling-serial-execution-across-your-organization.adoc[Controlling serial execution across your organization].

== Maintain version integrity with serial deployments

=== Goal

You may want to prevent older code from deploying after newer versions have already been deployed. Using serial groups with pipeline number priority ensures deployments happen in the correct order and automatically skips deployments from older pipelines.

=== Steps

. Add the `serial-group` key to your deployment job. The serial group should be specific to the deployment target:
+
[,yml]
----
workflows:
  deploy-workflow:
    jobs:
      - build
      - test
      - deploy-production:
          serial-group: production-deployment
          requires:
            - build
            - test
          filters:
            branches:
              only: main
----

. Use pipeline values to create environment-specific serial groups that prevent cross-environment deployment conflicts:
+
[,yml]
----
parameters:
  environment:
    type: string
    default: staging

workflows:
  deploy-workflow:
    jobs:
      - build
      - deploy:
          serial-group: << pipeline.parameters.environment >>-deployment
          requires:
            - build
----

. For multi-stage deployments, use different serial groups for each stage to allow parallel deployments to different environments:
+
[,yml]
----
workflows:
  multi-stage-deploy:
    jobs:
      - build
      - test
      - deploy-staging:
          serial-group: staging-deployment
          requires:
            - test
      - deploy-production:
          serial-group: production-deployment
          requires:
            - test
          filters:
            branches:
              only: main
----

=== Notes

* Pipeline number priority automatically skips jobs from older pipelines if a newer pipeline is running or waiting in the same serial group.
* This automatic skipping behavior helps maintain version integrity without manual intervention.
* Serial groups do not prevent you from starting new pipelines; they only control when jobs execute.
* You can view which pipelines are waiting in the queue by hovering over the serial group start block in the workflow graph.
* For more information, see xref:controlling-serial-execution-across-your-organization.adoc[Controlling serial execution across your organization].
