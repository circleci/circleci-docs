= Environment hierarchy and version promotion
:page-platform: Cloud
:page-description:
:experimental:

Environment hierarchy and version promotion let you define an ordered promotion path for your environments (for example, dev → staging → production) and visually promote a version through each stage from the CircleCI UI. Instead of manually tracking which version is deployed where, you get a single view of the entire release lifecycle with one-click promotion between environments.

== Key concepts

=== Environment hierarchy

An environment hierarchy is an ordered sequence of integrations that defines a promotion path. For example, a hierarchy could be:

* Development
* Staging
* Production

.Example environment hierarchy
[mermaid]
----
graph LR
    Dev[Development<br/>Environment] -->|Promote| Staging[Staging<br/>Environment]
    Staging -->|Promote| Production[Production<br/>Environment]
----

Hierarchies are configured at the organization level, with the option to override which hierarchy to use for a specific project or component. Each level in the hierarchy maps to one environment.

#screenshot showing the environment hierarchy example#

=== Version promotion

Version promotion is the act of deploying a specific version to the next environment in the hierarchy by triggering a deploy pipeline. The version promotion page provides a visual pipeline of environment cards so you can see the current state of every environment at a glance and promote forward with a single click.

== Prerequisites

Before using environment hierarchy and version promotion you need:

* *Environment integrations*. At least two integrations must exist for the organization (for example, a development environment and a production environment). You can create environment integrations in the CircleCI web app at menu:Deploys[Environments], select btn"[Create environment integration] and choose the custom option. See the xref:guides:deploy:configure-deploy-markers.adoc#create-an-environment-integration[Create an Environment Integration] page for more details. You can also choose to just configure your environment when creating your deploy markers which will automatically create an environment integration for you if it does not exist already.

* *Deploy markers* must be configured in your project to use a deploy pipeline. See the xref:guides:deploy:configure-deploy-markers.adoc[Configure Deploy Markers] page for more details.

* *Deploy pipeline*. The project must have a deploy pipeline configured. You can check this in menu:Project settings[Deploys], check the Deploy Pipeline section. See the xref:guides:deploy:set-up-deploys.adoc[Set up a Deploy Pipeline] page for more details.

== Set up an environment hierarchy and promote a version

Environment hierarchies are configured at the organization level. You can set multiple hierarchies at the organization level, set a default, and override the hierarchy for a specific project or component.

=== Create a hierarchy

. Navigate to menu:Org[Deploys] in the CircleCI web app.

. In the Environment Hierarchy card, select btn:[Create new hierarchy].

. Enter a name and optional description.

. Add levels by selecting from your existing environments. Each level maps to one environment.

. Order the levels to define the promotion sequence. The first level is the earliest environment (for example, dev) and the last level is the final environment (for example, production).

. You can set the hierarchy as the organization default. This step is optional.

=== Promote a version

. Select *Deploys* in the sidebar. you are now in your deployments timeline view. You can select a component to see a timeline for a specific component.

. Find the version you want to promote in the timeline list.

. Select the promote icon icon:promote[promote icon] to open the Version Promotion page if an environment hierarchy exists for the component. If no hierarchy exists, the Deploy Pipeline Modal opens directly.

== Manage hierarchies
=== Edit or delete a hierarchy

. Navigate to menu:Org[Deploys] in the CircleCI web app.

. Select the edit icon icon:edit-solid[Edit icon] for the hierarchy you want to edit, or the trash icon icon:delete[Delete icon] to delete the hierarchy.

. Edit the hierarchy by reordering levels, adding or removing integrations, updating the name or description.

. Select btn:[Update] to save your changes.

=== Assign a hierarchy to use for a project

=== Assign a hierarchy to use for a component

. Select *Deploys* in the sidebar.
. Find your component and select it by nae to enter the component view.
. Use the Icon:edit-solid[Edit icon] next ti *Hierarchy Template*.
. Select the hierarchy you wish to use in the *Replace with:* menu.
. Select btn:[Save].

=== Changing the hierarchy

Click the Change environment hierarchy button at the top of the page to navigate to Organization Settings > Deploys where you can modify or reassign the hierarchy.

=== Environment hierarchy assignment scopes

A hierarchy can be assigned at three scopes:

[.table-scroll]
--
[cols=2*, options="header"]
|===
| Scope | Behavior

| Organization
| Default hierarchy for all projects and components in the org

| Project
| Overrides the organization default for a specific project

| Component
| Overrides the project default for a specific component
|===
--

When the system resolves which hierarchy applies, it checks component first, then project, then organization. The most specific assignment wins.

== The version promotion page

#screenshot showing the version promotion page#

The version promotion page shows the following:

* Breadcrumbs — Releases Dashboard > Component Name > Version Promotion

* Metadata bar — project, component, version, commit SHA, author & message, and the name of the active promotion hierarchy

* Environment pipeline — a horizontal row of environment cards connected by action buttons (Promote, Retry, Promoting, Promoted)

=== Promote to the next environment

On the version promotion page, you can promote a version to the next environment in the hierarchy by clicking the Promote button between two environment cards. This opens the Deploy Pipeline Modal pre-filled with:

* The target environment (integration)
* The component
* The version being promoted

Review the values and click Deploy to trigger the deploy pipeline. A success toast confirms the pipeline was triggered, and the environment card updates to show the deployment in progress.

=== Retry a failed deployment

When a deployment fails:

* The environment card shows a red border and "Deployment failed" subtitle.

* A Retry button appears between the cards (or as an icon on the card itself).

* A link to the failed job is displayed for debugging.

Click Retry to re-trigger the deploy pipeline for that environment.


=== Environment card states

Each environment card in the promotion pipeline shows a status that reflects the current state of the deployment:

[.table-scroll]
--
[cols=4*, options="header"]
|===
| Status | Card Subtitle | Connector state | Description

| SUCCESS / LOGGED
| Currently deployed
| Promoted (checkmark)
| Version is live in this environment

| RUNNING / PENDING
| Deploying
| Promoting (spinner)
| Deploy pipeline is currently executing

| FAILED
| Deployment failed
| Retry (red button)
| Deploy pipeline failed; retry or investigate

|CANCELED
| Not deployed
| Retry (red button)
| Deploy was canceled; can retry

| Not deployed
| Not deployed
| Promote (blue button)
| Version has not been deployed to this environment
|===
--

Card details (visible when a release exists):

* Version — the version name being deployed (labeled "Attempted Version" on failure)

* Timestamp — "Deployed", "Started at", or "Failed at" depending on status

* Failed job link — direct link to the failed job (only on failure)
