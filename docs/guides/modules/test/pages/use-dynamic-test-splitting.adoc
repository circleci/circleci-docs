Static test splitting pre-assigns each node a fixed set of tests upfront. Dynamic test splitting distributes tests across parallel nodes so they finish at the same time, adjusting each node's workload based on when that node is ready to run tests.

== Is my project a good fit for dynamic test splitting?

=== Fast test runner startup times

Static test splitting runs all tests in a single batch. Dynamic test splitting runs tests across multiple batches.

If your test runner starts fast compared to the runtime of tests, the overhead of running multiple batches is negligible.

=== Nodes start at noticeably different times

Static test splitting pre-allocates tests to each node. Dynamic test splitting redistributes tests to nodes that are ready.

If your jobs have long container setup, dependency installation, or image pulls, each node's startup time can vary between runs. Dynamic test splitting moves tests from slow nodes to fast nodes.

=== JUnit test results

Static test splitting uses previous test results to preallocate tests to each node. Dynamic test splitting uses the same data to preallocate and redistribute tests.

If your job uses the `store_test_results` step, dynamic test splitting reads the previous JUnit runtimes on each run to preallocate and redistribute tests.

== How it works

When you configure parallelism in your job and enable dynamic test splitting, Smarter Testing automatically:

* Sorts tests by duration using timing data from previous runs, with longer tests prioritized first.
* Gives the first node that is ready a full batch of tests, like static test splitting.
* Gives each later-starting node a smaller batch sized so all nodes finish at the same time.
* Evenly distributes the remaining tests across all nodes.

== Prerequisites

Before enabling dynamic test splitting, ensure you have completed the xref:getting-started-with-smarter-testing.adoc[Getting Started With Smarter Testing] guide and have:

* Installed the `testsuite` CLI plugin.
* Configured your `.circleci/test-suites.yml` with `discover` and `run` commands.
* Verified your tests run successfully with the `testsuite` command.

== 1. Enable dynamic test splitting option in your test-suites.yml file

[source,yaml]
----
---
name: ci tests
# ...
options:
  dynamic-test-splitting: true
----

== 2. Run locally

Use `--doctor` locally to validate the `test-suites.yml` is set up correctly.

The CLI will run through additional checks when dynamic test splitting is enabled, if any results look incorrect an action
item is provided to resolve it. Follow the steps until all checks pass.

[source,console]
----
$ circleci run testsuite "ci tests" --doctor
----

[TIP]
====
Run the `circleci run testsuite` CLI tooling from the directory where your tests are located. This is typically your repository root, but for monorepos it can be the root of a subpackage (for example, `cd service-1 && circleci run testsuite "ci tests"`).

The `testsuite` tooling will automatically find the `.circleci/test-suites.yml` configuration file by walking up the directory tree. All commands (`discover`, `run`, `analysis`) execute relative to where you run the CLI, so avoid using `cd` or `--directory` flags within your commands.
====

== 3. Run in CI

Commit `.circleci/test-suites.yml` to your feature branch and push to your VCS.

Verify dynamic test splitting is working as expected in CI by looking at the "timings" tab in the UI.
